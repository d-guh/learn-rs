# Bringing Paths into Scope with the `use` Keyword
We can simplify paths with the `use` keyword to bring certain modules into scope. For example:
```rs
mod front_of_house {
    pub mod hosting {
        pub fn add_to_waitlist() {}
    }
}

use crate::front_of_house::hosting;

pub fn eat_at_restaurant() {
    hosting::add_to_waitlist();
}
```
`use` is similar to a symbolic link in filesystems.

Note that `use` only creates the shortcut in the same scope that the `use` occurs, the following example won't compile:
```rs
mod front_of_house {
    pub mod hosting {
        pub fn add_to_waitlist() {}
    }
}

use crate::front_of_house::hosting;

mod customer {
    pub fn eat_at_restaurant() {
        hosting::add_to_waitlist();
    }
}
```
## Creating Idiomatic `use` Paths
The previous example was the idiomatic way, since you have to call the parent module so it's clear that it is not locally defined.
The following example is also valid, but less clear:
```rs
mod front_of_house {
    pub mod hosting {
        pub fn add_to_waitlist() {}
    }
}

use crate::front_of_house::hosting::add_to_waitlist;

pub fn eat_at_restaurant() {
    add_to_waitlist();
}
```
With structs and enums, its idiomatic to specify the full path. This example shows how to bring the `HashMap` struct into scope:
```rs
use std::collections::HashMap;

fn main() {
    let mut map = HashMap::new();
    map.insert(1, 2);
}
```
An exception to this is bringing two items with the same name into scope with use requires using the parent module, since you cannot have 2 `Result`s for example:
```rs
use std::fmt;
use std::io;

fn function1() -> fmt::Result {
    // --snip--
}

fn function2() -> io::Result<()> {
    // --snip--
}
```
## Providing New Names with the `as` Keyword
Another solution to the problem of two types of the same name is the `as` keyword. This allows you to create a new local name/alias for the type:
```rs
use std::fmt::Result;
use std::io::Result as IoResult;

fn function1() -> Result {
    // --snip--
}

fn function2() -> IoResult<()> {
    // --snip--
}
```
## Re-exporting Names with `pub use`
When bringing a name into scope via `use` the name is private to the scope that imported it, to enable code outside of the scope to use it, we can combine `pub` and `use` in a technique called re-exporting.
```rs
mod front_of_house {
    pub mod hosting {
        pub fn add_to_waitlist() {}
    }
}

pub use crate::front_of_house::hosting;

pub fn eat_at_restaurant() {
    hosting::add_to_waitlist();
}
```
Before this, external code would have to use `restaurant::front_of_house::hosting::add_to_waitlist()`, which also required `front_of_house` to be public, but with this `pub use`, it has re-exported the hosting module from the root module, and external code can use the `restaurant::hosting::add_to_waitlist()` path instead.
## Using External Packages
To use external packages, you add the package names and versions to the `Cargo.toml` file:
```toml
rand = "0.8.5"
```
To bring the `rand` definitions into scope, we add a `use` line starting with the name of crate, and the items we want to bring into scope.
```rs
use rand::Rng;

fn main() {
    let secret_number = rand::thread_rng().gen_range(1..=100);
}
```
## Using Nested Paths to Clean Up `use` Lists
Rather than doing this:
```rs
// --snip--
use std::cmp::Ordering;
use std::io;
// --snip--
```
You can do this:
```rs
// --snip--
use std::{cmp::Ordering, io};
// --snip--
```
To import parent module:
```rs
use std::io::{self, Write};
```
## Importing Items with the Glob Operator
To bring all public items defined in a path into scope, use the GLOBBB `*`:
```rs
use std::collections::*;
```
